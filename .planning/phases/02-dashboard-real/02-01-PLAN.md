<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Supabase RLS | Queries são feitas com a chave publishable (anon key) — Row Level Security no Supabase deve garantir que empresa_id filtra corretamente |
| empresa_id vazio | Se useAuth() retornar tenant=null ou empresa.id='', queries ficam disabled (enabled: !!empresaId) — sem vazamento de dados |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | fetchDashboardMetrics | mitigate | Query sempre filtra por empresa_id — se RLS estiver ativo no Supabase, o filtro é duplo. Verificar RLS nas tabelas leads e agendamentos. |
| T-02-02 | Information Disclosure | empresaId vazio | mitigate | enabled: !!empresaId impede queries com string vazia. Implementado em DashboardPage.tsx. |
| T-02-03 | Spoofing | anon key hardcoded | accept | Chave publishable (anon) é projetada para uso público no browser. Segurança via RLS no Supabase, não via chave. |
| T-02-04 | Denial of Service | Promise.all sem timeout | accept | Dashboard usa Promise.all com 4 queries simultâneas — risco baixo para uso interno single-tenant. TanStack Query gerencia retry com retry: 1. |
</threat_model>
