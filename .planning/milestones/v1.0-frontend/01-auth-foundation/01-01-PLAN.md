## Threat Model

<threat_model>

### Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Supabase Auth | Email+password submitted from browser directly to Supabase HTTPS endpoint |
| Browser → FastAPI backend | Bearer JWT from Supabase sent in Authorization header |
| Browser → Supabase PostgREST | Supabase anon key + JWT used for table access |

### STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | LoginPage → signIn() | mitigate | Supabase Auth handles credential validation server-side; client never touches raw password hash |
| T-01-02 | Information Disclosure | supabase.ts hardcoded anon key | accept | Anon key is public by design (Supabase RLS enforces row-level access); not a secret |
| T-01-03 | Information Disclosure | JWT stored in localStorage | accept | Supabase default storage for browser; acceptable for v1 SPA; httpOnly cookies would require server-side proxy |
| T-01-04 | Elevation of Privilege | ProtectedRoute bypass | mitigate | ProtectedRoute checks `user` state from Supabase session on every render; session validated server-side on each Supabase/API call |
| T-01-05 | Spoofing | tenant.empresa.id from membros table | mitigate | `fetchTenant()` uses authenticated session's userId to query membros — user cannot inject a different empresa_id |
| T-01-06 | Denial of Service | Repeated failed logins | transfer | Supabase Auth handles rate limiting on auth endpoint |
| T-01-07 | Repudiation | No audit log for login events | accept | v1 scope; Supabase Auth logs available in dashboard |

</threat_model>
