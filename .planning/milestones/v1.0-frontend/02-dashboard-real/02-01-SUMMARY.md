# Phase 02 Plan 01: Dashboard Real Data Fixes Summary

**One-liner:** Corrected three bugs in dashboard data layer — wrong agendamentos date field, incorrect taxa de conversao formula, and missing explicit return type — with zero TypeScript errors after npm run build.

## What Was Changed

### leadflow-frontend-v2/src/services/api/dashboard.ts

**Correction 1 — fetchChartData: data_hora field**
The agendamentos table uses `data_hora` as its timestamp field, not `criado_em`. The query was selecting and filtering on `criado_em` which would return no data. Fixed both the `.select()`, `.gte()`, `.lt()` chain and the accumulation loop to use `data_hora`.

**Correction 2 — fetchDashboardMetrics: taxaConversao formula**
DASH-04 specifies taxa = COUNT(qualificado) / COUNT(total) * 100. The original query used `.in('status', ['qualificado', 'agendado', 'convertido'])` as the numerator — a funnel-progress count, not the DASH-04 definition. Changed to `.eq('status', 'qualificado')` only.

**Correction 3 — fetchRecentLeads: explicit return type**
Added `import type { Lead } from '@/types'` and changed the function signature to `Promise<Lead[]>` with `return (data ?? []) as Lead[]`. This satisfies TypeScript's strict checking and aligns with the exported interface contract.

### leadflow-frontend-v2/src/modules/dashboard/DashboardPage.tsx

**Correction 1 — Import Lead type**
Added `import type { Lead } from '@/types'` alongside the AuthContext import.

**Correction 2 — Remove inline type from map**
Replaced the verbose inline object type `(lead: { id: string; nome: string; telefone: string; status: string; score: number; criado_em: string })` with the canonical `(lead: Lead)`. The StatusBadge cast was retained as-is since StatusBadge's prop type is a narrower union.
